## Security

### 🟡 Medium — WatchControlListener accepts start/stop commands from any paired-device app without authentication

**File:** watch/sensor-capture/app/src/main/AndroidManifest.xml:58-64 / WatchControlListener.kt:10-32
**Lens:** Security
**Subsystem:** Watch

**What:** `WatchControlListener` is a `WearableListenerService` registered with `exported="true"` and an intent-filter for `com.google.android.gms.wearable.MESSAGE_RECEIVED`; any app on the paired phone that holds the `WEARABLE` permission can send a `/biosignals/start` or `/biosignals/stop` message and trigger or terminate a capture session.
**Why it matters:** A rogue or compromised phone app could remotely start an unbounded capture (`durationSec = -1`), draining the watch battery and collecting biometric data without the user's knowledge.
**Recommended fix:** Add a shared secret or nonce in the message payload (e.g., a session token generated by the phone app and verified by the watch), or restrict the path-filter to a custom permission declared in your app's manifest so only your own phone APK can send those messages.

### 🟡 Medium — No runtime check after permission grant before opening trackers

**File:** watch/sensor-capture/app/src/main/kotlin/com/littlesignals/capture/CaptureActivity.kt:116-128 / CaptureSession.kt:52-54
**Lens:** Security
**Subsystem:** Watch

**What:** `ensurePermissions()` fires the permission request dialog but `startCapture` / `CaptureSession.run()` proceed immediately when the user taps "Start" regardless of whether the dialog was acted upon yet, and there is no re-check of `checkSelfPermission` inside `CaptureSession` before calling `service.getHealthTracker()`.
**Why it matters:** If the user taps Start before granting permissions, the Samsung SDK opens trackers without the required runtime permissions, which can silently return null/empty data on SDK 1.4.x instead of throwing — leaving a session's CSV files empty with no visible error.
**Recommended fix:** In `CaptureActivity.startCapture`, check `ActivityCompat.checkSelfPermission` for each required permission and show an informative error (or re-trigger the permission flow) before invoking `CaptureSession.run()`.

### 🟢 Low — `isMinifyEnabled = false` in release build leaves class names and SDK calls fully visible

**File:** watch/sensor-capture/app/build.gradle.kts:23-25
**Lens:** Security
**Subsystem:** Watch

**What:** The release build type has `isMinifyEnabled = false`, meaning R8/ProGuard is disabled; all class, method, and field names are unobfuscated in the release APK.
**Why it matters:** An attacker who reverse-engineers the APK can trivially understand the data flow, path names (`/biosignals/start`), and the JSON schema used to communicate with the phone.

## Reliability

### 🔴 Critical — `SampleBatch` has no maximum size; a stalled phone will cause unbounded memory growth

**File:** watch/sensor-capture/app/src/main/kotlin/com/littlesignals/capture/SampleBatch.kt:17-36 / PhoneSenderConsumer.kt:59-64
**Lens:** Reliability
**Subsystem:** Watch

**What:** `SampleBatch` accumulates samples in four unbounded `mutableListOf` lists; `PhoneSenderConsumer.flushIfNonEmpty()` only drains them when `sender.send()` succeeds or when `stop()` is called. If the phone becomes unreachable (watch moves out of Bluetooth range, phone locks its radio), `WearPhoneSender.send()` returns `false` but the drain has already been called — meaning the samples in that batch are dropped. However, if the send is blocking (waiting on `Tasks.await`) and the phone node reappears and disappears repeatedly, or if batches are larger than expected, there is no ceiling on list size.
**Why it matters:** During an overnight sleep session (~8 hours), PPG at 25 Hz produces ~720,000 samples; without a bound the lists could exhaust the watch's heap (typically 256–512 MB shared across all apps), causing an OOM crash and losing all data.
**Recommended fix:** Add a `MAX_SAMPLES_PER_CHANNEL` constant (e.g., 500 for a 20-second safety margin) and drop or flush-early when the limit is hit, logging a warning.

### 🔴 Critical — Dropped samples on phone unreachability: `drain()` is called before confirming send succeeded

**File:** watch/sensor-capture/app/src/main/kotlin/com/littlesignals/capture/PhoneSenderConsumer.kt:59-64
**Lens:** Reliability
**Subsystem:** Watch

**What:** In `flushIfNonEmpty()`, `batch.drain()` clears the batch atomically and *then* calls `sender.send()`; if `send()` returns `false` (phone node unreachable), the drained samples are discarded with no retry and no local persistence.
**Why it matters:** Any Bluetooth gap during a sleep recording silently drops seconds or minutes of biometric data; the backend will see gaps in the time series without knowing whether they represent true absence of signal or a connectivity drop.
**Recommended fix:** Either (a) hold the `Drain` and retry with back-off before discarding, or (b) write samples to a local ring-buffer file and replay them when connectivity resumes, sending a `dropped_ms` counter in each batch payload so the backend can flag gaps.

### 🟠 High — `RemoteCaptureService` uses `START_NOT_STICKY`; if killed by Wear OS, the session silently ends

**File:** watch/sensor-capture/app/src/main/kotlin/com/littlesignals/capture/RemoteCaptureService.kt:43
**Lens:** Reliability
**Subsystem:** Watch

**What:** `onStartCommand` returns `START_NOT_STICKY`, meaning if the OS kills the service under memory pressure, it will not be restarted and no notification is sent to the phone that the session ended.
**Why it matters:** Wear OS aggressively kills background processes on watches without chargers; a session started remotely from the phone app can die silently overnight, with the phone never receiving a `/biosignals/end` message and waiting indefinitely.
**Recommended fix:** Return `START_REDELIVER_INTENT` (to restart with the original intent) or send a `/biosignals/end` with `reason = "service_killed"` via `onTaskRemoved`/`onDestroy`, so the phone side can surface a session-interrupted state to the user.

### 🟠 High — No reconnect logic if the Wearable Data Layer channel drops mid-session

**File:** watch/sensor-capture/app/src/main/kotlin/com/littlesignals/capture/WearPhoneSender.kt:20-34
**Lens:** Reliability
**Subsystem:** Watch

**What:** `WearPhoneSender.send()` calls `nodeClient.connectedNodes` on every send and simply returns `false` if the list is empty; there is no retry loop, no exponential back-off, and no in-memory queuing of failed sends.
**Why it matters:** A temporary Bluetooth hiccup (phone screen locked, watch moves to edge of BT range) silently drops every batch during the gap; for a 1 Hz flush rate that means data loss proportional to the gap duration.
**Recommended fix:** Implement a short retry loop (e.g., 3 attempts, 500 ms apart) inside `send()` before returning false, and/or buffer the last N failed batches in a `ConcurrentLinkedQueue` to replay when a node reappears.

### 🟠 High — `CaptureActivity` does not stop the session when the activity is backgrounded or destroyed

**File:** watch/sensor-capture/app/src/main/kotlin/com/littlesignals/capture/CaptureActivity.kt:87-138
**Lens:** Reliability
**Subsystem:** Watch

**What:** `CaptureActivity` starts a `CaptureSession` in a coroutine launched from a Compose `rememberCoroutineScope`; there is no `onStop()` or `onDestroy()` override that cancels the session. `FLAG_KEEP_SCREEN_ON` prevents normal ambient-mode dimming but does not prevent Wear OS from destroying the activity if the user presses the back or home button.
**Why it matters:** Pressing home during a session cancels the scope's coroutine (via Compose lifecycle), leaving the Samsung Health tracker listeners attached until garbage-collected (if the `HealthTracker` objects hold strong references), and the CSV writer may be left open mid-write, potentially corrupting the file.
**Recommended fix:** Override `onDestroy()` in `CaptureActivity`, cancel the capture coroutine explicitly, and ensure `CaptureSession.run()`'s `finally` block runs (it does run on coroutine cancellation, but only if the coroutine is actually cancelled — i.e., the scope must be cancelled).

### 🟡 Medium — `ChannelRecorder` objects are singletons (`object`); state leaks between capture sessions

**File:** watch/sensor-capture/app/src/main/kotlin/com/littlesignals/capture/ChannelRecorder.kt:116-203
**Lens:** Reliability
**Subsystem:** Watch

**What:** `HeartRate`, `PpgGreen`, `Eda`, and `Accelerometer` are Kotlin `object` declarations (singletons for the process lifetime). Their mutable state (`writer`, `tracker`, `sampleCount`, `firstSampleTsMs`, `lastSampleTsMs`) is never reset between calls to `start()`.
**Why it matters:** If the user taps "New capture" after a successful session, `sampleCount` and timestamp fields from the previous session contaminate the new session's `metadata.json`, reporting incorrect observed Hz values.
**Recommended fix:** Convert the channel implementations to regular `class` instances (allocated fresh by `all()`) or add an explicit `reset()` method called at the start of each session; `all()` already returns a new `listOf(...)` but the objects themselves are shared.

## Correctness & Data Integrity

### 🟠 High — `tStartMs` in serialized batch is `nowMs()` (wall clock at flush time), not the earliest sample timestamp

**File:** watch/sensor-capture/app/src/main/kotlin/com/littlesignals/capture/PhoneSenderConsumer.kt:62 / SampleBatch.kt:53
**Lens:** Correctness
**Subsystem:** Watch

**What:** `serializeBatchPayload(tStartMs = nowMs(), drain = drain)` uses the current system clock time as `tStartMs`, not the minimum `timestampMs` across the drained samples. Each sample already carries its own `t` field (the SDK `DataPoint.timestamp`), so `tStartMs` is redundant but potentially misleading: it represents when the flush coroutine ran, not when the first sample in the batch was collected.
**Why it matters:** If the phone backend uses `tStartMs` as an anchor for the batch (e.g., for deduplication or ordering), clock skew between `nowMs()` and the SDK timestamps (which are device-boot-relative on some versions) will produce incorrect relative timestamps; the `tStartMs` could be seconds after the last sample in the batch.
**Recommended fix:** Set `tStartMs` to `drain.hr.minOfOrNull { it.timestampMs } ?: nowMs()` (the true earliest sample timestamp across all channels), or remove `tStartMs` entirely and let the phone derive the batch window from the per-sample `t` fields.

### 🟡 Medium — Samsung SDK `DataPoint.timestamp` units are not verified; README says "UTC ms" but SDK convention varies

**File:** watch/sensor-capture/README.md:33 / ChannelRecorder.kt:77-79 / SampleBatch.kt:4
**Lens:** Correctness
**Subsystem:** Watch

**What:** The README states "`timestamp_ms` is the SDK's `DataPoint.timestamp` (wall-clock UTC milliseconds)." The Samsung Health Sensor SDK 1.4.x documentation describes `DataPoint.timestamp` as milliseconds elapsed since the Unix epoch, but historical versions used boot-relative nanoseconds or milliseconds. The code trusts this without any range-check or unit sanity assertion.
**Why it matters:** If a device returns boot-relative ms (which would be, e.g., 5,000–86,400,000 ms instead of ~1,700,000,000,000 ms), all timestamps stored in CSVs and sent to the phone would be nonsensical, corrupting every session silently.
**Recommended fix:** Add a one-time assertion at session start (or inside `firstSampleTsMs` setter) that checks the timestamp is within a plausible UTC range (e.g., `ts > 1_600_000_000_000L`) and logs a prominent error if not.

### 🟡 Medium — No schema version in the JSON batch payload; field additions will silently break the phone parser

**File:** watch/sensor-capture/app/src/main/kotlin/com/littlesignals/capture/SampleBatch.kt:49-51
**Lens:** Correctness
**Subsystem:** Watch

**What:** The batch JSON (fields: `tStartMs`, `hr`, `ppg`, `eda`, `accel`) has no `schemaVersion` field; the comment at line 49 warns "do not change keys without updating the phone side first" but there is no enforcement mechanism.
**Why it matters:** When new channels (e.g., skin temperature, SpO2) or new per-sample fields are added, there is no way for the phone parser to detect it is receiving data from a newer watch app and gracefully degrade; it will silently ignore new keys or fail if keys are renamed.
**Recommended fix:** Add `"schemaVersion": 1` to the root JSON object; increment on every breaking or additive change; have the phone parser reject or warn on unknown versions.

### 🟡 Medium — No batch sequence number; the phone cannot detect dropped or out-of-order batches

**File:** watch/sensor-capture/app/src/main/kotlin/com/littlesignals/capture/PhoneSenderConsumer.kt:38-64 / WearPhoneSender.kt:20-34
**Lens:** Correctness
**Subsystem:** Watch

**What:** Each `/biosignals/samples` message carries no sequence number or monotonic batch counter; the Wearable MessageClient does not guarantee delivery order between messages.
**Why it matters:** If two batches are reordered in transit (possible when BT reconnects), or if a batch is silently dropped (false-positive `true` return from the Wearable API), the backend time series will have an invisible gap or reversed segments with no way to detect or report it.
**Recommended fix:** Add a `batchSeq: Int` field (monotonically incrementing per session) to the batch payload; the phone can then detect gaps and log them as data quality events.

### 🟡 Medium — `isoUtc()` format string uses `HH-mm-ss` instead of `HH:mm:ss`; not a valid ISO 8601 timestamp

**File:** watch/sensor-capture/app/src/main/kotlin/com/littlesignals/capture/CaptureSession.kt:173-177
**Lens:** Correctness
**Subsystem:** Watch

**What:** `isoUtc()` formats as `"yyyy-MM-dd'T'HH-mm-ss'Z'"` (hyphens between time components) rather than the standard `"yyyy-MM-dd'T'HH:mm:ss'Z'"` (colons). The result (e.g., `2026-05-06T15-30-00Z`) is used both as the capture directory name and in `metadata.json` as `start_ts_utc` / `end_ts_utc`.
**Why it matters:** The directory naming using hyphens is intentional (colons are invalid in filenames), but writing the same hyphenated string into `metadata.json` as `start_ts_utc` means the value is not a valid ISO 8601 timestamp; any downstream tooling that parses it with a strict ISO 8601 parser will fail.
**Recommended fix:** Use two separate formatters: one with hyphens for the directory name, and one with colons (`HH:mm:ss`) for the JSON metadata.

### 🟡 Medium — Thread safety: `ChannelRecorder` singleton objects mutate `writer`, `tracker`, `sampleCount` from SDK callbacks without synchronization

**File:** watch/sensor-capture/app/src/main/kotlin/com/littlesignals/capture/ChannelRecorder.kt:31-104
**Lens:** Correctness
**Subsystem:** Watch

**What:** `writer`, `tracker`, `sampleCount`, `firstSampleTsMs`, and `lastSampleTsMs` are plain `var`/`Long?` fields on the singleton objects. `start()` writes `writer` and `tracker` on the caller thread; `onDataReceived` reads `writer` and writes `sampleCount`/`firstSampleTsMs`/`lastSampleTsMs` on the Samsung SDK callback thread; `stop()` nulls them on the caller thread. There are no locks or `@Volatile` annotations on these fields.
**Why it matters:** A race between `stop()` (setting `writer = null`) and a concurrent `onDataReceived` (calling `writer?.write(row)`) could cause an NPE or write to a closed stream if the SDK delivers a final callback after `stop()` is called; `sampleCount` increments without synchronization could also be lost.
**Recommended fix:** Mark `writer` and `tracker` as `@Volatile`, use `AtomicLong` for `sampleCount`, and guard `firstSampleTsMs`/`lastSampleTsMs` updates with `synchronized(this)` (consistent with how `SampleBatch` already uses `@Synchronized`).
